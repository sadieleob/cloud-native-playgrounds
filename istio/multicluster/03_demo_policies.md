# Multicluster Ambient Mesh - Waypoint & Authorization Policies

## Step 1: Deploy waypoint proxies

Waypoints handle L7 policy enforcement in ambient mode. Deploy one per cluster in the `nginx` namespace.

```bash
kubectl apply -f manifests/waypoint.yaml --context $CTX_EAST
kubectl apply -f manifests/waypoint.yaml --context $CTX_WEST
```

### Verify waypoints are programmed

```bash
kubectl get gateway waypoint -n nginx --context $CTX_EAST
kubectl get gateway waypoint -n nginx --context $CTX_WEST
```

Expected: `PROGRAMMED: True` with an assigned address.

## Step 2: Attach waypoints to nginx service

```bash
kubectl label svc nginx-service -n nginx --context $CTX_EAST istio.io/use-waypoint=waypoint
kubectl label svc nginx-service -n nginx --context $CTX_WEST istio.io/use-waypoint=waypoint
```

### Verify waypoint attachment

```bash
istioctl ztunnel-config service --context $CTX_EAST | grep nginx
```

Expected: Both `nginx-service` and `autogen.nginx.nginx-service` show `waypoint` / `autogen.nginx.waypoint` in the WAYPOINT column.

## Step 3: Apply AuthorizationPolicies

The policies target the waypoint Gateway (not the Service) to ensure enforcement for both local `svc.cluster.local` and cross-cluster `mesh.internal` traffic.

```bash
kubectl apply -f manifests/authz-policies.yaml --context $CTX_EAST
kubectl apply -f manifests/authz-policies.yaml --context $CTX_WEST
```

### Policy breakdown

**1. Default Deny** — No traffic reaches nginx unless explicitly allowed:
```yaml
apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: nginx-default-deny
  namespace: nginx
spec:
  targetRefs:
  - kind: Gateway
    group: gateway.networking.k8s.io
    name: waypoint
```

**2. Allow from `default` namespace** — Only workloads in the `default` namespace can access nginx:
```yaml
spec:
  targetRefs:
  - kind: Gateway
    group: gateway.networking.k8s.io
    name: waypoint
  action: ALLOW
  rules:
  - from:
    - source:
        namespaces: ["default"]
```

**3. Deny cross-cluster identity** — Explicitly deny traffic from west cluster's `restricted` namespace:
```yaml
spec:
  targetRefs:
  - kind: Gateway
    group: gateway.networking.k8s.io
    name: waypoint
  action: DENY
  rules:
  - from:
    - source:
        principals: ["demo-west.local/ns/restricted/sa/default"]
```

### Why target the Gateway, not the Service?

AuthorizationPolicies with `targetRefs` pointing to a Service only apply to traffic matching that Service's VIP. Cross-cluster traffic arrives on the autogenerated global service (`nginx-service.nginx.mesh.internal`) which is a different ServiceEntry. By targeting the waypoint Gateway directly, the policy applies to ALL traffic flowing through it — both local and cross-cluster.

## Step 4: Create restricted namespace for testing

```bash
for CTX in $CTX_EAST $CTX_WEST; do
  kubectl create ns restricted --context $CTX
  kubectl label ns restricted istio.io/dataplane-mode=ambient --context $CTX
  kubectl run curl-restricted -n restricted --context $CTX \
    --image=curlimages/curl:7.83.1 --restart=Never --command -- sleep infinity
done
```

## Step 5: Test AuthZ enforcement

### Test 1: Allowed — default namespace → nginx (local)

```bash
CURL_POD=$(kubectl get pod -l app=curl-client -n default --context $CTX_EAST -o jsonpath='{.items[0].metadata.name}')
kubectl -n default exec $CURL_POD --context $CTX_EAST -- curl -s -o /dev/null -w "HTTP %{http_code}" nginx-service.nginx.svc:8080
# Expected: HTTP 200
```

### Test 2: Denied — restricted namespace → nginx (local)

```bash
kubectl -n restricted exec curl-restricted --context $CTX_EAST -- curl -s -o /dev/null -w "HTTP %{http_code}" --max-time 5 nginx-service.nginx.svc:8080
# Expected: HTTP 403
```

### Test 3: Allowed — default namespace → mesh.internal (cross-cluster)

```bash
CURL_POD=$(kubectl get pod -l app=curl-client -n default --context $CTX_WEST -o jsonpath='{.items[0].metadata.name}')
for i in $(seq 1 6); do
  kubectl -n default exec $CURL_POD --context $CTX_WEST -- curl -s -w " | HTTP %{http_code}\n" --max-time 5 nginx-service.nginx.mesh.internal:8080
done
# Expected: HTTP 200 from both EAST and WEST clusters
```

### Test 4: Denied — restricted namespace → mesh.internal (cross-cluster)

```bash
for i in $(seq 1 6); do
  kubectl -n restricted exec curl-restricted --context $CTX_WEST -- curl -s -w " | HTTP %{http_code}\n" --max-time 5 nginx-service.nginx.mesh.internal:8080
done
# Expected: HTTP 403 (RBAC: access denied) from BOTH clusters
```

### Test 5: Allowed — default namespace → mesh.internal from east

```bash
CURL_POD=$(kubectl get pod -l app=curl-client -n default --context $CTX_EAST -o jsonpath='{.items[0].metadata.name}')
for i in $(seq 1 6); do
  kubectl -n default exec $CURL_POD --context $CTX_EAST -- curl -s -w " | HTTP %{http_code}\n" --max-time 5 nginx-service.nginx.mesh.internal:8080
done
# Expected: HTTP 200 from both EAST and WEST clusters
```

## Summary of capabilities demonstrated

| Capability | Evidence |
|---|---|
| **Automatic mTLS** | ztunnel logs show SPIFFE identities on every connection |
| **Cross-cluster service discovery** | `mesh.internal` hostname resolves to endpoints in both clusters |
| **Cross-cluster load balancing** | Requests distribute across east and west nginx instances |
| **L7 AuthorizationPolicy** | Waypoint enforces namespace-based allow/deny |
| **Cross-cluster identity enforcement** | Trust domain principals (`demo-west.local/ns/...`) used in deny rules |
| **Zero-trust default deny** | Only explicitly allowed traffic reaches the service |

## Mapping to Demo's architecture

| Demo requirement | Demo equivalent |
|---|---|
| Hub-spoke Azure regions (NA, EU, etc.) | Two Kind clusters simulating two regions |
| East-west microservice mTLS | Automatic ztunnel mTLS between all ambient workloads |
| Palo Alto firewall (north-south) | Waypoint + AuthZ (east-west L7 policy enforcement) |
| Multi-region service mesh | Solo peering with per-cluster trust domains |
| No remote API server access | Solo's push-based peering (not community Istio remote secrets) |
